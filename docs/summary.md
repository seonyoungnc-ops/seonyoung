# 게임 서비스 구매 한도 동시 결제 처리 설계

> **주제:** 1인당 구매 횟수 제한이 있는 아이템(예: item_A 최대 5개)에서 동시 결제 시도 시 처리 방안
> **결론:** 결제 진행 시점 Redis 락(방식 A) 권장

---

## 1. 문제 정의

커머스 / 게임 서비스에서 **1인당 구매 횟수 제한**이 있는 경우,
같은 유저가 **PC + Mobile 동시 접속** 환경에서 동시에 결제를 시도하면 Race Condition이 발생한다.

```
결제 Flow: 상점 진입 → 아이템 수량 선택 → 결제 시도 → 주문서 진입 → "결제 진행"
```

---

## 2. 처리 방안 비교

### 방안 1 — 사전 차단 (Pessimistic Lock)
**흐름:** 결제 요청 → Redis SET NX 락 획득 시도 → 선착순 1건만 통과 → 나머지 즉시 차단

**장점:** PG사 수수료 1회 / 환불 처리 없음 / 이중 청구 없음
**단점:** Redis 인프라 필요 / 락 서버 장애 시 결제 중단 위험

### 방안 2 — 사후 자동 취소 (Optimistic)
**흐름:** 모두 PG사 요청 → 복수 승인 발생 → 선착순 1건 확정 → 패자 건 취소/환불

**아이템 지급:** 최종 성공 1건에만 지급 (이중 지급 없음)

**장점:** 결제 진행 자체 단순 / 높은 동시성 처리 가능
**단점:** PG사 수수료 N회 / 환불까지 수 분~수일 / 보상 파이프라인 필수 / 전자금융거래법 위반 소지

---

## 3. 게임 서비스 특화 설계 — 3단계 방어선

### 1단계 — 클라이언트
- 서버 API로 잔여 한도 실시간 조회 (캐시 TTL 5~10s)
- 수량 선택 max = (한도 - 보유 수량) 동적 설정

### 2단계 — API / 멱등성 키
- 주문서 생성 시 서버에서 구매 횟수 실시간 재검증
- 고유 `idempotency_key` 발급 (동일 요청 재시도 중복 차단)

### 3단계 — DB 트랜잭션 (최종 안전망)
- PG사 승인 후 `SELECT FOR UPDATE`로 행 락
- 최종 한도 재검증 → 통과 시 INSERT + UPDATE

---

## 4. 핵심 설계 — 결제 진행 시점 락 (최종 권장)

**설계 원칙**
- **주문서 생성**: PC / Mobile 모두 자유롭게 허용 (한도 검증만)
- **결제 진행 버튼**: 이 시점에만 Redis 락으로 선착순 1건 처리

**Redis 구현**
```
lockKey = "pay_lock:{user_id}:{item_id}"

SET lockKey {order_id} NX EX 600
# NX: 없을 때만 설정 (원자적)
# EX 600: 10분 TTL

OK  → 락 획득 성공 → PG사 결제 진행
nil → "처리 중인 결제가 있어요" 안내

DEL lockKey  # 완료/실패/취소 시
```

**상태별 처리**

| 상황 | 처리 |
|---|---|
| PC가 먼저 결제 진행 | 락 획득 → 정상 결제 |
| Mobile이 이후 시도 | 차단 + 주문서 유효 유지 |
| PC 결제 완료 | 락 해제 → Mobile 재시도 가능 (기존 주문서 재사용) |
| PC 결제 실패/취소 | 락 해제 → PC 또는 Mobile 재시도 가능 |
| 앱 강제 종료 | TTL 만료 자동 해제 |

---

## 5. Steam 비교 — 왜 같은 선상에서 보면 안 되는가

| 구조적 차이 | Steam | 이번 케이스 |
|---|---|---|
| 재고 구조 | 무한 재고 (디지털 복사본) | 유한 재고 (1인당 최대 5개) |
| 중복 결제 시 충돌 | 없음 | Race Condition 발생 |
| 동시 승인 허용 | 가능 (어떤 건이 먼저 처리되든 동일) | 불가 (한도 초과 차감 발생) |
| 패자 개념 | 없음 (모두 성공) | 반드시 발생 |
| 한도 정책 | 없음 | 비즈니스 핵심 정책 |

> **핵심 차이:**
> Steam = "자원이 무한해서 충돌이 없는 것"
> 방식 B = "자원이 유한한데 충돌을 허용한 뒤 사후 수습"

---

## 6. 최종 권장 구조

```
상점 진입     → 잔여 한도 표시, 수량 선택 max 제한       [클라이언트]
주문서 생성   → 한도 검증만 — 락 없음, 자유롭게 허용      [서버]
결제 진행     → pay_lock 획득 경쟁 — 선착순 1건만 통과    [서버 Redis] ← 핵심
DB 기록       → SELECT FOR UPDATE 최종 안전망             [DB]
```

"주문서는 자유, 결제는 선착순 1건" — UX와 안전성을 동시에 만족하는 설계.
